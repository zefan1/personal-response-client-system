# 跨平台前台窗口识图设计

## 背景

桌面端当前使用 Electron `desktopCapturer` 截取主屏幕，并把整块桌面交给识图服务。该方式会把辅助系统、终端和其他窗口同时包含在图片中，真实识图模型可能无法从复杂桌面中提取聊天消息。

产品不仅需要支持微信和企业微信，还需要支持抖音网页后台及后续其他聊天平台，因此不能依赖窗口标题、平台名称或固定聊天区域。用户也不接受首次框选或手动维护裁剪区域。

## 目标

- 用户点击一次“识别”即可完成截图和识图，不增加框选或配置步骤。
- 支持微信、企业微信、抖音网页后台、Chrome、Edge 及其他前台聊天窗口。
- 捕获完整前台窗口，不物理裁剪聊天区域，由识图 LLM 自行定位当前主会话。
- 不使用进程注入、内存读取、UI 自动化或高频窗口监听。
- 捕获或识图失败时自动进入回复助手并显示具体失败原因。

## 非目标

- 不自动点击、输入或发送聊天消息。
- 不读取微信、企业微信、浏览器或抖音的内部接口、数据库或进程内存。
- 不保存截图到磁盘、数据库、业务日志或长期缓存。
- 不在本阶段实现用户手动框选、平台专用插件或固定坐标模板。

## 方案选择

### 方案 A：继续截取整个桌面

实现简单，但桌面噪声会持续干扰识图，无法解决已复现的问题，不采用。

### 方案 B：自动裁剪聊天区域

可以减少视觉噪声，但存在误裁关键消息、客户信息或上下文的风险，且用户明确不接受该风险，不采用。

### 方案 C：完整前台窗口直接识图

辅助系统短暂隐藏后，获取操作系统当前恢复到前台的目标窗口，截取该窗口完整画面，并一次性发送给识图 LLM。该方案不绑定平台、不丢失窗口内上下文，作为本次推荐方案。

## 架构

### Electron 主进程

新增独立的前台窗口捕获协调器，职责包括：

1. 记录辅助系统窗口当前可见、焦点和置顶状态。
2. 隐藏辅助系统窗口。
3. 等待操作系统把先前使用的聊天窗口恢复为前台，最长等待 500 毫秒，使用条件轮询而不是固定长延迟。
4. 使用 `get-windows` 读取前台窗口标识、进程、标题和边界；窗口信息只在当前捕获调用内使用。
5. 使用 Electron `desktopCapturer` 的 `window` 来源优先按窗口标识匹配前台窗口；标识不可直接对应时，才在当前调用内使用标题、进程和边界辅助匹配，然后按窗口边界生成完整窗口截图。
6. 无法匹配窗口来源时，降级为隐藏辅助系统后的当前屏幕截图。
7. 无论成功或失败，都恢复辅助系统原有的可见、焦点和置顶状态。

捕获结果增加非敏感元数据：

```ts
type ScreenshotResult = {
  success: boolean;
  imageBase64?: string;
  width?: number;
  height?: number;
  captureMode?: 'FOREGROUND_WINDOW' | 'SCREEN_FALLBACK';
  error?: string;
  message?: string;
};
```

窗口标题、进程名和窗口标识不返回 renderer、不写入日志或持久化存储。

### Renderer

现有蓝色“识别”入口继续调用统一的 `captureScreenshot()`。页面在识别开始时立即进入回复助手，显示截图和识图进度。renderer 不参与窗口选择，也不接触窗口标题或进程信息。

捕获失败时复用现有失败状态；捕获成功后把 `imageBase64` 交给现有 `triggerRecognize()`，不改变桌面端业务 API 路径。

### 识图后端

识图请求仍调用现有 OpenAI-compatible 多模态接口，但提示词改为平台无关：

- 判断当前画面是否包含正在使用的主聊天会话。
- 支持微信、企业微信、抖音网页后台和未知聊天平台。
- 忽略联系人列表、会话列表、导航栏、浏览器工具栏、订单信息和其他非当前会话内容。
- 保留完整窗口上下文，不要求模型返回裁剪坐标。
- 根据消息气泡位置、视觉样式和上下文区分客户与管家；无法确定时不得猜测。
- 手机号为可选字段，昵称、平台账号或其他可见客户标识可以作为匹配线索。
- 严格返回 JSON。

建议的模型返回结构：

```json
{
  "status": "OK",
  "platform": "DOUYIN_WEB",
  "nickname": "客户昵称",
  "phone": null,
  "customerIdentifier": "平台账号或其他可见标识",
  "messages": [
    { "role": "client", "text": "消息内容" },
    { "role": "keeper", "text": "回复内容" }
  ],
  "timestamp": null,
  "confidence": 0.92,
  "failureReason": null
}
```

无法识别时返回：

```json
{
  "status": "UNABLE_TO_DETERMINE",
  "platform": "UNKNOWN",
  "nickname": null,
  "phone": null,
  "customerIdentifier": null,
  "messages": [],
  "timestamp": null,
  "confidence": 0,
  "failureReason": "当前窗口未显示可识别的主聊天会话"
}
```

后端解析器对现有返回结构保持兼容；新字段为可选。`status=UNABLE_TO_DETERMINE` 或空消息列表时，使用 `failureReason` 生成安全、可操作的 `30-10001` 错误信息。

## 数据流

```text
用户点击识别
  -> 回复助手显示识别中
  -> 主进程隐藏辅助系统
  -> 获取并截取完整前台窗口
     -> 匹配失败时截取隐藏后的当前屏幕
  -> 恢复辅助系统
  -> 单次调用识图 LLM
  -> 解析平台、客户标识和消息
  -> 回复助手显示建议或具体失败原因
```

## 安全边界

- 每次捕获必须由用户主动点击触发。
- 一次点击最多产生一次窗口截图和一次默认识图请求。
- 不持续枚举、监听或录制目标窗口。
- 不注入目标进程，不使用 DLL Hook、内存读取或可访问性树抓取聊天内容。
- 不模拟鼠标、键盘、点击、输入或发送消息。
- 截图只存在于内存和当前 HTTP 请求内，请求结束后释放引用。
- 应用日志只记录捕获模式、尺寸、耗时和错误码，不记录截图、窗口标题、进程名、客户标识或聊天内容。

## 错误处理

- 未找到前台窗口：尝试屏幕降级捕获。
- 前台窗口来源匹配失败：尝试屏幕降级捕获。
- 捕获结果为空：恢复辅助系统并返回 `CAPTURE_FAILED`。
- 识图返回非法 JSON：返回现有图片识别失败错误，并显示可重试和文字通道。
- 识图明确无法判断：显示模型提供的安全失败原因。
- 辅助系统恢复失败：记录非敏感错误并强制显示主窗口，不影响后续人工操作。

## 测试策略

### 自动化测试

- 主进程单元测试：隐藏、条件等待、前台窗口匹配、窗口截图、屏幕降级及 `finally` 恢复。
- preload/bridge 测试：新捕获元数据正确传递，不暴露窗口标题或进程信息。
- renderer 测试：识别开始立即进入回复助手；捕获失败、识图失败和超时状态可见。
- 后端测试：兼容旧 JSON；解析微信、企业微信、抖音网页后台示例；处理可选手机号、客户标识、未知平台和 `UNABLE_TO_DETERMINE`。
- renderer smoke：Electron 窗口隐藏后能够恢复，捕获失败不会导致窗口消失或置顶状态丢失。

### 人工验收

- 微信桌面端当前会话。
- 企业微信桌面端当前会话。
- Chrome 和 Edge 中的抖音网页后台。
- 浏览器存在侧栏、订单区和多个会话列表时，只提取当前主会话。
- 双屏、窗口非最大化、辅助系统置顶和目标窗口最小化后的恢复场景。
- 非聊天页面应明确失败，不得生成虚假客户或消息。

## 发布边界

本设计只证明本地代码和人工场景可用。正式发布前仍需要在目标 Windows 版本、实际微信/企业微信版本、Chrome/Edge 版本和抖音后台页面上完成真实验收；不得用模拟图片或单一平台测试替代跨平台验收。
