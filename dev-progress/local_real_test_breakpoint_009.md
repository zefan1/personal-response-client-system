# 断点 009：客户档案搜索重复、旧门店显示与保存后自动刷新

时间：2026-07-09 14:05-14:18  
状态：已完成

## 本轮反馈

- 搜索 `1111` 出现两个客户。
- 明细里已经把意向门店改成 `万江店`，但搜索结果/顶部仍显示旧门店。
- 保存后还需要手动点刷新，期望保存成功后自动刷新。

## 根因

- 本地 smoke 数据库里残留了一条自动化负向用例脏数据：`phone=****1111`，所以搜索 `1111` 会出现第二个空客户。
- 多结果搜索后打开客户档案时，前端没有清理旧搜索结果，导致上方搜索卡片继续展示旧门店。
- 客户带 `sourceRowId` 时，保存成功后会先弹“是否同步企微表格”，原逻辑没有立即回读最新档案，导致需要手动刷新确认。

## 处理结果

- 已删除本地 smoke 库中的 `****1111` 脏客户，并清理 Redis 客户缓存。
- 后端搜索增加防线：
  - `CustomerRepository.searchByNickname/searchByKeyword` 排除带 `*` 的手机号。
  - `CustomerSearchService` 返回前只保留 11 位真实手机号客户。
- 前端客户档案增加刷新链路：
  - 新搜索开始时清空旧搜索结果。
  - 打开客户档案时清空旧搜索结果，避免旧卡片挡在详情上方。
  - 保存成功后先自动重新拉取详情，再保留“是否同步表格”的提示。
- 已用真实接口把 `18800001111` 的意向门店保存为 `万江店`，并确认详情接口和搜索接口都立即返回 `万江店`。

## 验证结果

- 前端定向单测通过：`npm run test -- src/renderer/modules/customer-profile/customerProfileStore.test.ts`，11 个用例通过。
- 前端类型检查通过：`npm run typecheck`。
- 后端定向测试通过：`mvn -Dtest=CustomerSearchServiceTest,CustomerControllerTest test`，10 个用例通过。
- 真实接口验证：
  - `GET /api/v1/customers/search?q=1111&limit=10` 返回 `total=1`。
  - 返回客户为 `18800001111 / 188****1111`，`intendedStore=万江店`。
  - `GET /api/v1/customers/18800001111` 返回 `intendedStore=万江店`。

## 当前可手工验证

- 刷新或重开侧边栏，进入客户档案。
- 搜索 `1111`，预期只出现/自动打开 `逾期跟进客户 / 188****1111` 一条。
- 点击“编辑档案”，修改“意向门店”等字段并保存，预期保存后详情自动显示新值，不需要再点右上角刷新。
- 如果弹出“是否同步到企微表格”，这是额外的表格同步确认；档案本身已经先保存并自动刷新。
