# 标签功能完整修复执行方案 055

时间：2026-07-13  
状态：关键决定已经确定，可以按步骤直接执行  
适用要求：这是生产系统，所有修改必须完成数据库、后端、管理后台、侧边栏、Skill、LLM、规则、统计和测试的完整检查

## 1. 为什么取消员工逐条确认

之前的方案要求员工确认、拒绝或修改每一条 AI 标签，这会增加员工工作量，也会让自动分析失去意义。

现在固定改成：

1. 系统判断清楚并且通过后端检查后，自动更新客户标签。
2. 系统判断不清楚时，不修改客户当前标签。
3. 系统不能为了得到结果强行选择最接近的标签。
4. 员工不需要逐条确认 AI 判断。
5. 员工可以随时手动新增、修改或移除客户标签。
6. 员工手动修改后，该分类默认锁定，系统不能继续自动覆盖。
7. 员工可以主动解除锁定，让系统以后继续自动更新。
8. 每次自动判断和人工修改都要留下记录，方便主管追查。

本次只取消“标签”的逐条确认。

客户档案中其他非标签字段，例如预约时间、到店时间等，如果现有流程需要员工确认，继续保留原来的确认方式。本次不能顺便删除整个客户档案建议功能。

## 2. 标签最终要解决什么问题

标签有四个实际用途：

1. 员工打开客户档案时，可以快速了解客户当前情况。
2. Skill 生成回复时，可以读取客户当前标签，调整回复方向。
3. 主管可以按标签筛选客户，例如查看中意向客户数量。
4. 跟进规则可以根据标签安排提醒和优先级。

最终流程固定为：

    聊天记录 + 当前客户档案 + 当前有效标签
                        ↓
              Skill 说明分析任务
                        ↓
         LLM 根据后台标签说明进行判断
                        ↓
     后端检查分类、标签、证据、权限和当前数据
                        ↓
        判断清楚则自动更新，判断不清楚则不修改
                        ↓
     员工可以随时手动新增、修改、移除或锁定
                        ↓
        回复、筛选、统计和跟进规则读取最新标签

## 3. 后续开发不得再改变的决定

除非用户明确提出新要求，否则后续 AI 必须遵守：

1. 管理后台保存的标签配置是唯一标准。
2. Skill 和直接调用 LLM 必须读取同一份标签配置。
3. Skill、LLM 和后端不能分别写死不同标签。
4. 现有四类标签只是初始默认数据，不是系统固定上限。
5. 有权限的主管或管理员可以新增、修改、停用和合并标签分类。
6. 有权限的主管或管理员可以新增、修改、停用和合并具体标签。
7. 组员可以给自己有权限的客户新增、修改和移除已有标签。
8. 组员默认不能修改全公司的标签分类，除非账号权限明确允许。
9. 新增分类或标签后，不需要修改代码、增加数据库字段或重新发版。
10. 系统内部编码由后端自动生成，页面不要求运营人员填写英文编码。
11. 内部编码生成后不能修改；中文名称、说明和例子可以修改。
12. LLM 只能返回当前存在、已启用并允许系统判断的标签。
13. LLM 可以返回无法判断，无法判断时不修改客户标签。
14. 自动更新失败不能影响正常回复生成。
15. 人工修改优先于系统自动判断。
16. 当前有效标签才用于回复、筛选、统计和跟进规则。
17. 已停用标签可以保留在历史客户中，但不能再分配给新客户。
18. 已被客户使用的标签不能直接删除，只能停用或合并。
19. 所有数据库变更必须通过启动时自动执行的数据库迁移完成。
20. 改完重启后必须立即可用，不允许要求用户另外执行补数据脚本。

## 4. 初始四类标签怎么处理

现有四类先保留：

- 性格类型
- 身体关注
- 客户顾虑
- 意向等级

这些是数据库中的默认分类，后端代码不能通过分类名称写死特殊判断。每个分类如何判断、是否自动更新、单选还是多选，都从数据库配置读取。

### 4.1 性格类型

用途：帮助 Skill 调整沟通方式。

默认设置：

- 单选。
- 允许系统自动判断。
- 需要较多有效聊天内容。
- 判断不清楚时保持原值或待判断。
- 员工可以手动修改并锁定。

边界：

- 不能把一次情绪变化当成长期性格。
- 客户这一次着急，不代表一定是果断型。
- 聊天内容不足时不能自动写入。
- 系统不能根据年龄、地区等无关信息推断性格。

### 4.2 身体关注

用途：记录客户明确关注的问题。

默认设置：

- 多选。
- 允许系统自动添加。
- 第一版系统只能自动添加，不能自动删除已有身体关注标签。
- 员工可以手动添加或移除。

边界：

- 只能记录客户明确表达的内容。
- 不能进行医学诊断。
- 客户没有提到的问题不能依靠常识推断。
- 无法对应现有标签时不修改，并记录为未匹配结果供主管查看。

### 4.3 客户顾虑

用途：记录影响客户决策的原因。

默认设置：

- 多选。
- 允许系统自动添加。
- 第一版系统只能自动添加，不能根据一次聊天自动删除旧顾虑。
- 员工可以手动移除已经解决的顾虑。

边界：

- 客户明确说“太贵了”，可以添加担心价格高。
- 客户只说“我再看看”，不能直接判断为担心价格高。
- 只靠语气猜测时不修改标签。
- 同一客户可以同时存在多个顾虑。

### 4.4 意向等级

用途：安排跟进优先级和查看客户数量。

默认设置：

- 单选。
- 允许系统自动更新。
- 需要防止短时间内频繁变化。
- 员工可以手动修改并锁定。

边界：

- 明确预约、提出到店时间可以提高意向。
- 只问一次价格不能直接设为高意向。
- 已成交必须来自真实成交数据、外部同步结果或员工手动设置。
- 已流失必须符合后台配置的明确规则或员工手动设置。
- 单条消极聊天不能立即把高意向降为低意向。

## 5. 主管可以新增和修改什么

### 5.1 新增或修改分类

分类表单必须包含：

- 分类名称。
- 分类用途。
- 单选或多选。
- 是否允许系统自动判断。
- 是否允许员工手动修改。
- 系统判断清楚后是自动添加、自动替换还是只记录结果。
- 系统自动更新需要达到的最低把握程度。
- 至少需要多少条有效客户消息。
- 自动更新后多久内不能再次变化。
- 判断不清楚时保持原值还是设置为待判断。
- 是否用于回复生成。
- 是否用于客户筛选。
- 是否用于统计。
- 是否允许跟进规则使用。
- 是否启用。
- 显示顺序。

新增分类时：

- 后端自动生成内部分类编码。
- 不增加 customers 表的新字段。
- 分类数据保存到统一客户标签记录中。
- 保存成功后，Skill、LLM、筛选和规则自动读取。
- 不需要重新部署。

### 5.2 新增或修改具体标签

标签表单必须包含：

- 中文名称。
- 标签含义。
- 什么情况下应该使用。
- 什么情况下不能使用。
- 正确例子。
- 错误例子。
- 常见同义说法。
- 是否允许系统自动选择。
- 是否允许员工手动选择。
- 是否启用。
- 显示顺序。

内部标签编码：

- 由后端自动生成。
- 页面可以显示“系统编号”，但不能要求运营人员输入英文。
- 创建后不能修改。
- 所有数据库关联、规则和接口都使用这个编号。
- 中文名称改变时，历史客户和规则不能失效。

### 5.3 停用、删除和合并

- 没有任何客户、规则和历史记录使用时，才允许删除。
- 已经被使用时只能停用或合并。
- 停用后不能用于新的自动判断和手动分配。
- 历史客户继续显示原中文名称。
- 合并前显示受影响客户、跟进规则和统计数量。
- 合并必须自动更新客户标签、规则引用和统计使用的编号。
- 合并完成后旧编号保留历史对应关系，不能直接消失。

## 6. 组员怎么手动修改客户标签

客户档案中的标签区域需要支持：

- 查看当前有效标签。
- 查看标签来源：系统判断、员工修改、业务数据、历史导入。
- 查看系统判断依据。
- 添加一个已有标签。
- 修改单选分类的当前标签。
- 给多选分类增加或移除标签。
- 锁定某个分类，阻止系统继续自动修改。
- 解除锁定，允许系统以后继续更新。

权限要求：

- 组员只能修改自己有权限查看的客户。
- 组长可以修改自己团队范围内的客户。
- 管理员按现有权限范围处理。
- 没有全局标签管理权限的组员不能新增公司级分类或标签。

手动修改保存后：

- 后端立即保存。
- 当前页面自动刷新。
- 其他已登录页面通过现有实时通知自动更新。
- 写入操作记录。
- 后续回复立即使用新标签。
- 系统自动判断不能覆盖锁定分类。

## 7. 客户标签在数据库中怎么保存

必须先检查真实数据库结构，再新增数据库迁移。

需要新增统一的客户标签记录，不再依赖给每个新分类增加一个 customers 表字段。

每条记录至少保存：

- 客户唯一标识。
- 分类唯一标识。
- 标签唯一标识。
- 当前是否有效。
- 来源：系统判断、员工修改、业务规则、外部导入。
- 系统判断的把握程度。
- 客户原话或业务数据依据。
- 使用的 Skill。
- 使用的 LLM 环境和模型。
- 使用的提示词版本。
- 创建时间和更新时间。
- 操作账号。
- 是否人工锁定。
- 失效原因。

多选分类可以有多条当前有效记录。

单选分类同一时间只能有一条当前有效记录。保存新值时，必须在同一次数据库操作中让旧值失效并保存新值，不能出现短暂重复。

旧 customers 表中的 personality_type、body_concerns、worries、intent_level 等字段：

- 在迁移期间继续维护，避免现有功能断开。
- 新客户标签记录和旧字段必须在同一次保存中同步更新。
- 所有读取位置完成切换前不能停止更新旧字段。
- 新增自定义分类只写统一客户标签记录，不给 customers 表新增字段。
- 最终所有筛选、统计、回复和规则都读取统一客户标签记录。

## 8. 历史数据必须怎样处理

启动时自动执行数据库迁移，不允许用户手动运行脚本。

迁移规则固定为：

1. 旧值与内部编码完全一致时，自动生成客户标签记录。
2. 旧值与当前中文名称完全一致时，转换成对应内部编码并生成记录。
3. 多值字段必须识别项目当前真实使用的分隔方式。
4. 无法识别的原始文字不能猜测。
5. 无法识别值要完整保存到“历史未匹配标签记录”，页面可查看原始内容。
6. 无法识别值不参与正式筛选、统计和 Skill 判断。
7. 迁移前后客户总数和原始内容不能减少。
8. 数据库迁移重复执行不能生成重复记录。
9. 迁移完成后系统可以立即运行，不要求人工补数据。

管理后台需要提供历史未匹配值页面：

- 查看原始值。
- 查看来自哪个客户和哪个旧字段。
- 选择映射到已有标签。
- 新增合适标签后再映射。
- 批量处理完全相同的原始值。
- 所有处理记录进入操作日志。

这不是上线前必须手工完成的步骤。未匹配值保留原文且不影响系统运行。

## 9. Skill 和 LLM 应该收到什么

Skill 负责说明任务，标签内容必须由后端动态提供。

每次分析需要发送：

- 最近有效聊天记录。
- 当前客户档案。
- 当前有效标签。
- 已人工锁定的分类。
- 本次允许系统判断的分类。
- 每个分类的单选或多选要求。
- 每个候选标签的名称、含义、使用条件、禁止条件和例子。
- 每个分类的自动更新要求。

LLM 必须返回：

- 分类内部编号。
- 标签内部编号。
- 对该判断的把握程度。
- 能给员工查看的客户原话或业务依据。
- 结果类型：可以更新、无法判断、保持当前值。
- 多选分类需要说明是新增标签还是保持不变。

禁止返回：

- 后台不存在的标签。
- 已停用标签。
- 不属于目标分类的标签。
- 详细内部思考过程。
- 医学诊断。
- 没有依据的强制判断。

Skill 和直接 LLM 两条路径必须使用相同内容和相同返回要求。

## 10. 后端怎么决定是否自动更新

后端收到结果后按下面顺序检查：

1. 客户存在。
2. 当前账号或系统任务有权处理该客户。
3. 分类存在并启用。
4. 分类允许系统自动判断。
5. 标签存在、启用并属于该分类。
6. 标签允许系统自动选择。
7. 单选或多选数量符合分类设置。
8. 客户原话或业务依据不为空。
9. 有效客户消息数量达到分类设置。
10. 把握程度达到分类设置。
11. 分类没有被员工锁定。
12. 距离上次自动更新的时间符合分类设置。
13. 客户数据版本没有被其他操作更新。

全部通过后：

- 单选分类按分类设置自动替换或保持。
- 多选分类第一版只自动新增，不自动删除。
- 保存客户标签记录。
- 同步旧客户字段。
- 写入操作记录。
- 发送实时更新通知。

任何一项不通过：

- 不修改当前标签。
- 记录具体中文原因。
- 保留本次分析记录，供主管查看系统为什么没有更新。
- 不影响回复功能。

## 11. 人工修改和系统自动更新发生冲突时怎么办

固定优先顺序：

    人工锁定结果
        高于真实业务数据以外的所有自动判断

    真实业务数据
        高于 LLM 判断

    LLM 判断
        只在分类未锁定并满足后台设置时更新

具体规则：

- 员工手动修改分类后自动锁定。
- 锁定分类不接收新的 LLM 更新。
- 员工解除锁定后，下一次聊天分析才可以更新。
- 已成交等真实业务结果不能被 LLM 改回未成交状态。
- 外部数据同步和员工同时修改时，使用现有客户版本检查，失败方重新读取最新数据。
- 所有覆盖、拒绝和冲突都写入操作记录。

## 12. 回复生成需要同步修改

生成回复前，后端需要读取：

- 客户当前有效标签。
- 标签中文名称和含义。
- 标签来源。
- 客户阶段。
- 最近跟进记录。

Skill 和直接 LLM 都必须获得同一份当前标签。

必须同步修改：

- 侧边栏生成回复接口。
- 后端聊天处理服务。
- Skill 回复请求。
- 直接 LLM 回复请求。
- 换一组回复保存的上下文。
- 在线测试页面。
- 回复调用记录。
- 回复相关自动测试。

回复要求：

- 标签只用于调整方向和语气。
- 不能在回复中直接告诉客户“系统把你判断为某种类型”。
- 标签读取失败时继续使用普通客户档案生成回复。
- 标签读取失败不能返回空白回复或导致页面卡住。

## 13. 管理后台搜索、统计和规则需要同步修改

### 客户搜索

需要支持：

- 按分类筛选。
- 按一个或多个标签筛选。
- 单选分类按一个当前值筛选。
- 多选分类支持包含任一标签或同时包含全部标签。
- 标签筛选和现有姓名、手机号、门店、项目、来源、管家筛选可以组合。
- 保持账号数据权限隔离。
- 支持分页和导出。

### 统计

至少显示：

- 每个当前有效标签的客户数量。
- 按门店、团队、员工和时间查看数量。
- 系统自动添加数量。
- 员工手动添加或修改数量。
- 系统判断但没有更新的数量和原因。
- 标签变化趋势。

统计规则：

- 只统计当前有效标签。
- 历史失效标签不计入当前数量。
- 历史未匹配原始值不计入正式数量。
- 权限范围必须与客户列表一致。

### 跟进规则

- 创建规则时从数据库读取当前启用标签。
- 保存规则时检查分类和标签仍然存在。
- 规则执行时只读取当前有效标签。
- 标签停用后，相关规则显示警告并暂停新的命中。
- 标签合并时自动更新规则引用。
- 规则列表、详情、创建、编辑、启停、删除、筛选和统计全部测试。

### 导入、同步和导出

- 导入优先识别内部编号。
- 中文名称完全一致时可以自动转换。
- 无法识别值保存到历史未匹配记录。
- 导出同时提供中文名称和内部编号。
- 外部表格同步失败不能删除或覆盖本地有效标签。

## 17. 禁止出现的做法

- 禁止把四类默认标签写死成系统只能支持四类。
- 禁止新增分类时要求修改 customers 表。
- 禁止要求运营人员填写英文内部编号。
- 禁止在 Skill、LLM 和后端维护不同标签列表。
- 禁止只改页面不改数据库和后端。
- 禁止只改后端不检查管理后台和侧边栏。
- 禁止相信 Skill 或 LLM 返回值一定正确。
- 禁止 LLM 判断不清楚时强制选择。
- 禁止系统自动删除多选标签。
- 禁止系统覆盖人工锁定标签。
- 禁止根据关键词猜测无法识别的历史数据。
- 禁止把历史未匹配值算入正式统计。
- 禁止使用前端空值处理掩盖后端字段缺失。
- 禁止只测试当前客户或当前四类标签。
- 禁止修改完成后再要求用户手动补数据。
- 禁止未完成全量测试就开启真实使用。
